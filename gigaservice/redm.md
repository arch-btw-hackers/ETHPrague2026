Архитектурный план: Tracker Analyze Service
Данный документ описывает архитектуру сервиса аналитики трекера (Service 2) для первой фазы проекта. Сервис принимает телеметрию, анализирует ее на соответствие заданным условиям, сохраняет данные и предоставляет API для внешних систем.
1. Роль и задачи сервиса (Phase 1)
Прием телеметрии: Получение данных от аппаратного трекера (Hardware Tracker) по HTTP.
Хранение данных: Сохранение истории перемещений и показаний сенсоров в Swarm.
Анализ условий: Проверка входящих данных на соответствие условиям доставки (задаются при создании контракта).
Предоставление данных для UI: API для фронтенда (дашборда), чтобы отображать маршрут и состояние сенсоров.
Управление данными (CRUD): Базовые операции создания, чтения, обновления и удаления записей.
2. Основные компоненты
Сервис можно логически разделить на следующие модули:
2.1. API Gateway / Ingestion Layer  #done
HTTP Endpoint: Принимает POST-запросы от трекера с данными (координаты, температура, влажность, удары и т.д.).
Валидация: Проверка формата и подписи данных (Signed telemetry).
2.2. Processing & Rules Engine (Анализатор)
Загрузка условий: При старте доставки сервис получает или запрашивает условия из смарт-контракта или базы данных (например: температура не выше 5°C, запрет ударов > 2G).
Анализ в реальном времени: Сверка каждого входящего пакета данных с условиями.
Определение статуса: "Все хорошо" (продолжаем) или "Все плохо" (нарушение условий).
2.3. Data Storage (База данных: Swarm) #done
Хранение данных децентрализовано.
Swarm: Используется как основное хранилище для истории телеметрии и информации о доставках.
CRUD Операции: Реализация логики взаимодействия со Swarm для записи, чтения и управления данными.
2.4. CRUD API Module
REST API для управления сущностями (Трекеры, Условия, Доставки).
Позволяет фронтенду и другим сервисам взаимодействовать с данными.
2.5. Dashboard API (Для Frontend)
REST или GraphQL API, которое фронтенд запрашивает для построения графиков и карты на дашборде.
Эндпоинты: получение текущего статуса, истории координат, графиков сенсоров. Чтение данных происходит из Swarm.
3. Взаимодействие (Data Flow)
Инициализация: Фронтенд инициирует доставку. Tracker Service получает ID трекера и условия доставки (через CRUD API), сохраняет их в Swarm.
Движение: Трекер отправляет HTTP POST с JSON: {"tracker_id": "123", "lat":..., "lon":..., "temp": 4, "timestamp":...}.
Обработка:
Ingestion Layer принимает данные и отправляет их в Swarm.
Rules Engine проверяет: temp (4) < max_temp (5) -> OK.
Нарушение (Упс!):
Трекер присылает temp: 10.
Rules Engine фиксирует нарушение.
(Дальнейшая логика уведомления других систем о нарушении будет добавлена позже).
Завершение: Трекер достигает пункта назначения. Записывается финальный статус в Swarm.
4. Стек технологий (Рекомендация)
Язык/Фреймворк: Node.js (NestJS или Express) / Go / Python (FastAPI).
Хранилище: Swarm (Bee client API).
Инфраструктура: Docker, Docker Compose (для локальной разработки).
5. Что пока "за бортом" (Phase 2 & 3)
Blockchain Integration: Прямое взаимодействие со смарт-контрактами вынесено из этого сервиса.
ENS: Интеграция с ENS сервисом отложена.
WebSockets: Как и было сказано, пока используем только HTTP(s) polling для дашборда и обычные POST запросы от трекера.
